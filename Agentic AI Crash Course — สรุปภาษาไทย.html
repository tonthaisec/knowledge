บทที่ 1
AI Agent คืออะไร กันแน่?
สรุปสั้น: AI Agent ไม่ใช่แค่ Chatbot ที่ตอบคำถาม — มันเป็นระบบที่ คิดเอง ตัดสินใจเอง และลงมือทำจริง
💬

ChatGPT (2022) — จุดเปลี่ยน

เปิดยุค AI สนทนาโดยไม่ต้องเขียนโค้ด ไม่ต้องเทรนโมเดลเอง ใครๆ ก็ใช้ได้

🚀

2024 — ผู้คนอยากได้มากกว่านั้น

แค่ตอบคำถามไม่พอแล้ว ต้องการให้ AI จัดการ email, วิเคราะห์ข้อมูล, ทำรายงานได้เอง

🛠️

Agent ทำงานอย่างไร?

ใช้ Tools (API, function call) เป็น "มือ" และ LLM เป็น "สมอง" ตัดสินใจว่าจะใช้ Tool ไหน เมื่อไหร่

🗂️

สูตรสำเร็จ Agent

LLM + Tools + Planning + Memory = ระบบที่ทำงานจากต้นจนจบได้โดยอัตโนมัติ

💡 Insight สำคัญ: Agent ไม่ใช่การค้นพบ AI ใหม่ — มันคือ "วิศวกรรมห่อหุ้ม" รอบๆ โมเดล AI ที่มีอยู่แล้ว ประเด็นอยู่ที่ ออกแบบให้ดีแค่ไหน
🗂️
บทที่ 2
4 ประเภทของ Agentic Systems
หัวใจ: เลือก Agent ให้เหมาะกับงาน ไม่ใช่เลือกตามความล้ำของเทคโนโลยี
⚙️ Rule-Based
อัตโนมัติต่ำ — กฎตายตัว
📋 Workflow Agent
ช่วยมนุษย์ตัดสินใจ
🤝 Semi-Autonomous
ทำเองได้ บางจุดหยุดถาม
🚁 Autonomous
อัตโนมัติเต็ม — เป้าหมายกว้างๆ
⚙️

1. Rule-Based Systems

if-this-then-that แบบดั้งเดิม ไม่มี AI จริงๆ เร็ว ตรวจสอบง่าย แต่เปลี่ยนยาก

ตัวอย่าง: อนุมัติค่าใช้จ่ายอัตโนมัติ, เปลี่ยนชื่อไฟล์

📋

2. Workflow Agents

LLM ช่วยทำงานในกระบวนการที่มีอยู่แล้ว แต่ มนุษย์ยังตัดสินใจสุดท้าย

ตัวอย่าง: Draft ตอบ email, สรุปประชุม, แปลง query สำหรับ BI

🤝

3. Semi-Autonomous Agents

วางแผนหลายขั้นตอน ใช้ Tools และทำงานเองได้ มี checkpoint ให้มนุษย์ตรวจ

ตัวอย่าง: ติดตาม lead, จัดการเอกสาร, Research agent

🚁

4. Autonomous Agents

รับเป้าหมายกว้างๆ แล้วคิดเองทำเอง ทำงานข้ามระบบโดยไม่ต้องสั่ง ความเสี่ยงสูง

ตัวอย่าง: Competitive research, Autonomous testing

🎯 Framework ตัดสินใจ: ถามว่า "งานมีโครงสร้างชัดแค่ไหน? ต้องการ Language ไหม? ใครตัดสินใจ?" ก่อนเลือก Agent ประเภท
🔧
บทที่ 3
Tools ใน AI คืออะไร?
สรุป: Tools คือ "มือ" ของ Agent — ทำให้ LLM ออกไปทำอะไรได้จริงในโลก ไม่ใช่แค่พูด
🔗

Tools คืออะไร?

ความสามารถภายนอกที่ LLM เรียกใช้ได้ — API, database queries, internal services, code functions ที่เชื่อมกับโลกจริง

🎯

ทำไมต้องมี Tools?

LLM โดยธรรมชาติ ไม่รู้ข้อมูล realtime และ ทำอะไรไม่ได้จริง Tools แก้ปัญหานี้

🔄 วงจรการทำงาน (Agent Loop)

👤 User Request → 🧠 LLM วางแผน → 📞 เรียก Tool → ⚙️ Execute → 👁️ Observe → 🔁 วนซ้ำจนเสร็จ
✅

ประโยชน์ 4 อย่าง

① Unlock Execution — ทำจริงไม่ใช่แนะนำ
② เพิ่ม Precision — ดึงข้อมูลจริง
③ Control Risk — Developer กำหนดขอบเขตได้
④ Composability — เชื่อม CRM, Email, Calendar

📝

ออกแบบ Tool ที่ดี

ต้องมี: ชื่อชัดเจน, คำอธิบาย "ใช้เมื่อไหร่", input parameters แบบ typed, output ที่ structured ให้ LLM ต่อได้

📚
บทที่ 4
RAG และ Agentic RAG คืออะไร?
RAG = Retrieval-Augmented Generation = ให้ AI ดึงข้อมูลจริงขององค์กรก่อนตอบ แทนที่จะอ้างอิงแค่ความรู้ตอน Train
🔍 แบบไหน📋 วิธีทำงาน✅ เหมาะกับ
Traditional RAG Query → ดึงข้อมูลครั้งเดียว → ตอบ คำถามง่ายๆ เช่น Policy lookup
Agentic RAG ดึงข้อมูลหลายรอบ ระหว่างทำงาน ปรับแผนได้ งานซับซ้อน multi-step
🏢

ทำไม Enterprise ชอบ RAG?

~70% ของ Enterprise GenAI use case ใช้ RAG เพราะ ข้อมูลองค์กรเปลี่ยนตลอด, Fine-tune แพง, RAG เร็วกว่า ปลอดภัยกว่า ตรวจสอบได้

🔄

Agentic RAG ต่างกันอย่างไร?

ไม่ใช่ดึงข้อมูลแค่ครั้งเดียว แต่ดึงซ้ำระหว่างงาน ตรวจสอบข้อเท็จจริง ลด hallucination และปรับ context ได้ realtime

📂 Data Ingestion → ✂️ Chunking & Index → 🔍 Retrieve → 💬 Context Injection → 🤖 Generate
🔌
บทที่ 5
MCP คืออะไร? และทำไมต้องสนใจ?
MCP (Model Context Protocol) = มาตรฐานกลางสำหรับ "จัดแพ็กเกจ" ข้อมูลทุกอย่างให้ LLM — เหมือน HTTP สำหรับเว็บ, แต่ใช้กับ AI
📦

MCP รวมอะไรไว้ด้วยกัน?

Task instructions + Tools & APIs + Retrieved documents + Memory + Conversation history → ส่งให้ LLM เป็นแพ็กเกจเดียว

🏗️

ทำไม Enterprise ต้องสนใจ?

ระบบ Agent ใหญ่ขึ้น → จัดการ Tools, Memory, RAG พร้อมกันยากมาก MCP ทำให้ modular และ composable ได้

⚡

ทำไม Adoption เร็ว?

ออกแบบมาสำหรับ AI โดยเฉพาะ, Anthropic ทำ docs + SDK ดีมาก, Network effect หลังปี 2025

🎯

ใครควรใช้ MCP?

Enterprise-grade agents, Multi-tool workflows, ระบบที่ context management เป็น bottleneck — ไม่จำเป็นสำหรับ prototype ง่ายๆ

💡 Key point: MCP ไม่ได้ทำให้โมเดลฉลาดขึ้น — แต่ทำให้ข้อมูลที่ส่งให้โมเดล มีโครงสร้างและสะอาดกว่า
🗺️
บทที่ 6
Planning ใน Agents และ Reasoning Models
ปัญหา: LLM ถูก Train ให้ "ทำนาย token ถัดไป" ไม่ได้ Train ให้วางแผนหลายขั้นตอน จึงมักหลงทาง ข้ามขั้น หรือลืมเป้าหมาย
โมเดลวิธีทำงานปัญหา
LLM ทั่วไปInput → Output ทันทีวางแผนซับซ้อนไม่เก่ง
LRM (Reasoning Model)Input → วางแผน → Outputช้า แพง บางที overthink
🧩

Planning ครอบคลุมอะไร?

แปล instructions → แบ่ง subtasks → ดึงข้อมูล → เลือก Tools → ลำดับขั้นตอน → จัดการ error → รู้ว่าเสร็จเมื่อไหร่

🤔

Reasoning Models คืออะไร?

Train ให้ "คิดก่อนตอบ" — o-series (OpenAI), DeepSeek-R1, Gemini Thinking, Claude reasoning mode

⚠️ ข้อควรระวัง: Reasoning models ดีขึ้น แต่ก็ overthink งานง่ายๆ, output ยาวกว่า, latency สูงกว่า และอาจ hallucinate แผนที่ฟังดูสมเหตุสมผลแต่ผิด

คำแนะนำ: เริ่มด้วย base model ก่อน เปลี่ยนเป็น reasoning model เมื่อเห็นปัญหา planning จริงๆ เท่านั้น
🧠
บทที่ 7
Memory ใน AI Agents
สภาพจริง: LLM โดย default ไม่มี memory — ทุก input ถือว่าเป็นอะไรใหม่ ต้องออกแบบ Memory ใส่ไปเองเสมอ
ประเภทขอบเขตเนื้อหา
🔵 Short-Term Memoryเฉพาะ session นั้นประวัติสนทนา, Tool calls, เอกสารที่ดึงมา
🟣 Long-Term Memoryข้าม session ได้User preferences, งานที่ทำไปแล้ว, identity
📖
Semantic Memory
ข้อเท็จจริงเกี่ยวกับ user
ตัวอย่าง: "ชอบรับไฟล์เป็น Excel"
📜
Episodic Memory
บันทึกเหตุการณ์ที่ผ่านมา
ตัวอย่าง: "สรุปรายงานให้เมื่อวาน"
⚙️
Procedural Memory
ความชอบส่วนตัว, Style guide
ตัวอย่าง: "หลีกเลี่ยง passive voice"
🎯 หลักการ: "Problem-first เสมอ" — Memory ที่ดีไม่ใช่เก็บให้มากที่สุด แต่เก็บ สิ่งที่ถูกต้อง ดึงได้ทันเวลา
🌐
บทที่ 8
Multi-Agent Systems
แนวคิด: แทนที่จะมี Agent ตัวเดียวทำทุกอย่าง แบ่งเป็น Agent หลายตัวที่เชี่ยวชาญแต่ละด้าน ทำงานพร้อมกัน
👑

Hierarchical (มีผู้นำ)

Orchestrator สั่งงาน Sub-agents ที่มีหน้าที่เฉพาะ เหมาะกับงานที่แบ่งงานชัดเจน ต้องการ governance เข้ม

ใช้กับ: Enterprise workflows, parallel processing

🤝

Flat (เท่าเทียมกัน)

Agent คุยกันโดยตรง ไม่มีหัวหน้า เหมาะกับการ brainstorm หรืองานที่ไม่มีคำตอบเดียว

ใช้กับ: Creative problem-solving, multi-perspective analysis

⚠️ ความเสี่ยงที่มองข้าม: ยิ่ง Agent มาก → ยิ่ง non-deterministic → ความผิดพลาดทบทวีคูณ, latency สูง, cost พุ่ง, และ "Collusion" (Agent เห็นด้วยกันโดยไม่ได้คิด)

คำแนะนำ: เริ่มด้วย Single Agent ที่ดีก่อนเสมอ 70%+ ของ Enterprise use case ไม่ต้องการ Multi-Agent
🌍
บทที่ 9
ระบบ Agentic AI ในโลกจริง
เรียนรู้จากสิ่งที่มีอยู่แล้ว: 3 ระบบจริงที่ใช้หลักการ Agentic AI ในระดับต่างกัน
📓

Google NotebookLM

Research assistant สำหรับ document Q&A ดึงข้อมูลและสังเคราะห์จากหลายไฟล์

ระดับ Autonomy: 2-3
มนุษย์ยังชี้นำ

🔍

Perplexity

ค้นหาเว็บ → ตัดสินใจใช้อะไร → สังเคราะห์คำตอบพร้อม citation วน loop หลายรอบ

ระดับ Autonomy: 3-4
อิสระมากขึ้น

🧪

OpenAI DeepResearch

รับเป้าหมายกว้างๆ แบ่ง subtasks เอง ทำซ้ำจนเสร็จ minimal human guidance

ระดับ Autonomy: 4
อัตโนมัติเต็มที่

💡 Insight: ทั้ง 3 ระบบใช้ส่วนประกอบเดียวกัน: Planning, RAG, Memory, Tools — ต่างกันแค่ว่า มนุษย์ควบคุมมากแค่ไหน

บทที่ 10
บทเรียนและอนาคตของ Agentic AI
บทสรุปคอร์ส: Agent = ระบบตัดสินใจ ไม่ใช่แค่ Chatbot | มีหลายระดับ | ต้องวางแผน memory, tools, RAG อย่างตั้งใจ

⚠️ สิ่งที่ทีมมักมองข้าม

  • 1ออกแบบ Observability ตั้งแต่วันแรก — track tool calls, latency, decision chains อย่ารอให้มีปัญหาก่อน
  • 2อย่าอยู่ใน "PoC Purgatory" — ทดสอบแบบไม่เป็นระบบ ใช้ metrics จริง: task completion, hallucination rate, token efficiency
  • 3Start Simple — Agent ตัวเดียวที่ดีมักดีกว่า Multi-Agent ที่ซับซ้อนโดยไม่จำเป็น

🔮 อนาคตของ Agentic AI

📏
Standards Over Improvisation
MCP และ Google A2A จะกลายเป็น standard วิธีส่งข้อมูลระหว่าง Agent กับ LLM
🎛️
Selective Reasoning
โมเดลจะเลือกได้ว่าเมื่อไหรควร "คิดลึก" และเมื่อไหรตอบเร็วพอ
🧠
Sophisticated Memory
จาก "ต่อกันด้วยกาว" สู่ memory ที่ scope ตามงาน session และ persona ได้จริง
🏗️
Mature Tooling
จาก custom wrappers สู่ trusted APIs มาตรฐาน พร้อม security practices ที่แข็งแรง
🏆 บทเรียนสุดท้าย (และสำคัญที่สุด):
Problem-first เสมอ อย่าเริ่มจากเทคโนโลยี เริ่มจากปัญหาที่อยากแก้ Start simple, measure rigorously, scale deliberately.
